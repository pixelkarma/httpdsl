⚡ httpdsl vs Native Go — Benchmark

wrk: 4 threads, 50 connections, 10 seconds per test

Testhttpdsl (req/s)Go (req/s)RatioBefore → After
Simple JSON 72,587 80,437 0.90x 69,836 → 72,587 (+4%)
Path Params + String 68,518 77,657 0.88x 67,819 → 68,518 (+1%)
JSON Parse + Echo 55,823 63,344 0.88x 57,310 → 55,823 (~same)
Fibonacci(20) sequence 13,860 69,766 0.20x 9,780 → 13,860 (+42%)
100-element array build 29,317 65,310 0.45x 5,033 → 29,317 (+483%)
Optimizations applied:
  • Skip mutex on per-request environments (only global scope locks)
  • Singleton NULL/BREAK/CONTINUE to eliminate allocation per statement
  • Reuse loop environment in each instead of allocating per iteration
  • Inline x = append(x, val) pattern — bypasses function dispatch entirely
  • Fast path for i += 1 (int64 + int64 without interface boxing)
  • Pre-sized environment maps for function calls
Results: API workloads (JSON in/out) sit at 88-90% of native Go. Computation went from 7-14x slower to 2-5x slower. Array building saw the biggest gain: 5,033 → 29,317 rps (+483%) from inlining append.

Remaining gap: The 2-5x computation overhead is inherent to tree-walking interpretation: AST dispatch, interface{} boxing, map-based variable lookup. Closing this further requires either bytecode compilation or JIT — both viable next steps.
